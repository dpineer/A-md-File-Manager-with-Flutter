LM-Studio 协议说明书（OpenAI 兼容 HTTP SSE 接口）
====================================================

一、概述
-------

LM-Studio 提供与 OpenAI API 完全兼容的 HTTP 接口，支持流式（SSE）和
非流式两种响应模式。本说明文档详细描述其协议规范，适用于替换 Dify
后端为 LM-Studio 本地推理引擎的场景。

二、网络端点
----------

基础 URL: http://{host}:{port}/v1
对话接口: POST /v1/chat/completions

默认主机: localhost
默认端口: 1234

完整地址示例: http://localhost:1234/v1/chat/completions

三、请求头
---------

Content-Type: application/json
Authorization: Bearer {API_Key} (LM-Studio 本地模式下可省略或传占位符)

四、请求体 JSON 结构
-------------------

{
  "model": "string",           // 模型名称（可选，LM-Studio 忽略该字段）
  "messages": [                // 消息列表（必填）
    {
      "role": "string",        // 角色：user / assistant / system
      "content": "string"      // 消息内容
    }
  ],
  "stream": true,              // 是否启用流式响应（建议 true）
  "temperature": 0.7,          // 温度参数，范围 0.0 - 2.0（可选，默认 0.7）
  "max_tokens": 4096,          // 最大生成长度（可选，默认 4096）
  "top_p": 1.0,                // 核采样参数（可选，默认 1.0）
  "frequency_penalty": 0.0,    // 频率惩罚（可选，默认 0.0）
  "presence_penalty": 0.0      // 存在惩罚（可选，默认 0.0）
}

字段说明:
- model:       模型标识符，LM-Studio 本地模式下该字段会被忽略，
               模型由用户在 LM-Studio 界面中加载的模型决定
- messages:    对话消息列表，按时间顺序排列，每条消息包含 role
               （user/assistant/system）和 content（消息文本）
- stream:      是否流式输出。true 表示 SSE 流式，false 表示一次性返回
- temperature: 控制输出的随机性。值越高输出越随机，越低越确定
- max_tokens:  限制生成的最大 token 数量
- top_p:       核采样参数，控制采样范围
- frequency_penalty: 频率惩罚系数，减少重复词的出现
- presence_penalty:  存在惩罚系数，鼓励讨论新话题

五、流式响应 SSE 格式
-------------------

接口会将每个响应片段以 SSE 格式逐行推送。每个事件以 "data: " 开头，
以双换行符结束。当 stream: false 时，返回标准 JSON 体。

每个 SSE 事件行格式:

data: {JSON 数据}

事件中的 JSON 数据格式:

{
  "id": "string",                    // 响应唯一标识符
  "object": "chat.completion.chunk", // 对象类型，固定值
  "created": 1234567890,             // 创建时间戳
  "model": "string",                 // 模型名称
  "choices": [                       // 选择列表
    {
      "index": 0,                    // 选择序号
      "delta": {                     // 增量数据
        "content": "string"          // 本次响应的文本片段
      },
      "finish_reason": null          // 结束原因（null 表示未结束）
    }
  ]
}

字段说明:
- id:           本次响应的会话唯一标识
- object:       固定为 "chat.completion.chunk"
- created:      Unix 时间戳（整型）
- model:        当前使用的模型名称
- choices:      选择列表，通常只有一个元素
  - index:       该 choice 的下标
  - delta:       增量数据，包含本次 SSE chunk 中的新文本
    - content:   实际的文本增量，为空字符串时表示无新内容
  - finish_reason: 结束原因
    - null:      未结束
    - "stop":    模型主动结束
    - "length":  达到 max_tokens 限制结束
    - "content_filter": 内容过滤触发结束

流式结束信号:

data: [DONE]

收到 [DONE] 信号表示整个流式响应已发送完毕。

非流式响应（stream: false）格式:

{
  "id": "string",
  "object": "chat.completion",
  "created": 1234567890,
  "model": "string",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "string"          // 完整响应内容
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {                         // token 用量（可选）
    "prompt_tokens": 0,              // 提示所用 tokens
    "completion_tokens": 0,          // 生成所用 tokens
    "total_tokens": 0                // 总 tokens
  }
}

六、流式解析规则
--------------

客户端应按如下流程解析 SSE 流:

1. 从字节流中累积数据，按换行符 '\n' 切分
2. 对每一行:
   a. 去除首尾空白
   b. 跳过空行
   c. 检测是否以 "data: " 开头:
      - 是: 提取 "data: " 后的内容
        - 若内容为 "[DONE]": 表示流结束，停止解析
        - 否则: 进行 JSON 解析
      - 否: 跳过该行
3. 从解析后的 JSON 中提取:
   choices[0].delta.content 即为本次收到的文本增量
4. 将所有 delta.content 拼接起来即为完整响应
5. 检查 choices[0].finish_reason:
   - 为 "stop" 或 "length": 模型响应结束
   - 为 null: 继续接收

七、完整请求示例
--------------

请求:

POST /v1/chat/completions
Host: localhost:1234
Content-Type: application/json
Authorization: Bearer lm-studio

{
  "messages": [
    {"role": "user", "content": "请解释什么是量子计算"}
  ],
  "stream": true,
  "temperature": 0.7,
  "max_tokens": 4096
}

流式响应示例:

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"content":"量子"},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"content":"计算"},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"content":"是一"},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"content":"种..."},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","created":1699999999,"model":"local-model","choices":[{"index":0,"delta":{"content":""},"finish_reason":"stop"}]}

data: [DONE]

说明: 首个 chunk 的 delta 中可能包含 "role" 字段（固定为 "assistant"），
不包含 "content" 字段，客户端应跳过该 chunk。最后一个带内容的 chunk
之后会有一个 finish_reason 为 "stop" 的空内容 chunk，之后是 [DONE]
结束信号。

八、错误处理
----------

当请求格式错误或服务异常时，服务端返回以下格式:

HTTP 状态码: 400 (请求错误) / 401 (认证失败) / 500 (服务端错误)

{
  "error": {
    "message": "错误描述信息",
    "type": "error_type",
    "code": 400
  }
}

常见错误:
- 400: 请求体 JSON 格式错误，或 messages 字段缺失/格式不对
- 401: Authorization header 格式错误（如无 Bearer 前缀）
- 422: messages 内容不合法（如空 content）
- 500: 服务端内部错误，模型未加载或推理异常

九、与 Dify API 的关键区别
------------------------

1. 端点不同:
   - Dify:      POST /v1/chat-messages
   - LM-Studio: POST /v1/chat/completions

2. 请求体不同:
   - Dify:      {inputs, query, response_mode, user, messages}
   - LM-Studio: {messages, stream, temperature, max_tokens}

3. 鉴权方式:
   - Dify:      Authorization: Bearer {app_secret}
   - LM-Studio: 本地模式无需鉴权

4. 消息格式:
   - Dify:      query 为单条消息，messages 为历史
   - LM-Studio: messages 为完整的对话列表，含当前消息

5. SSE 解析不同:
   - Dify:      data: {"answer": "文本片段"}
   - LM-Studio: data: {"choices":[{"delta":{"content":"文本片段"}}]}

6. 结束信号:
   - Dify:      读取完最后一个 data 即结束
   - LM-Studio: 以 data: [DONE] 显式结束
